# Сценарии использования

Возможные пользовательские флоу приложения — что делает пользователь, что происходит под
капотом и что он видит. Формат: **предусловия → шаги → ожидаемо**. Дополняется по мере роста
функциональности. Связано: [ARCH.md](ARCH.md) (как это устроено), [CHANGELOG.md](CHANGELOG.md)
(когда появилось).

## 1. Онбординг

### 1.1 Регистрация нового пользователя

- **Предусловия:** пользователь не залогинен, открыл `/register`.
- **Шаги:** вводит имя пользователя и пароль (≥ 8 симв.), опц. отображаемое имя → «Зарегистрироваться».
- **Ожидаемо:** бэкенд создаёт пользователя (argon2id-хеш), сразу выполняется вход (см. 1.2),
  редирект на ленту. Ошибки (занятое имя, короткий пароль) показываются под формой без перезагрузки.
- **Негатив:** ≥ 5 регистраций/час с одного IP → `429 Too Many Requests`.

### 1.2 Вход

- **Предусловия:** есть аккаунт, открыт `/login`.
- **Шаги:** имя + пароль → «Войти».
- **Ожидаемо:** access-токен в память, refresh — в httpOnly-cookie; WS-соединение открывается с
  токеном в `connectionParams`; персист кэша включается; редирект на ленту.
- **Негатив:** неверные данные → ошибка под формой (тайминг-безопасно, без утечки «есть ли юзер»);
  ≥ 10 попыток/15 мин → `429`.

### 1.3 Тихое восстановление сессии при перезагрузке

- **Предусловия:** пользователь был залогинен, обновляет страницу (F5).
- **Шаги:** — (автоматически на старте).
- **Ожидаемо:** access-токен потерян, но refresh-cookie жив → bootstrap тихо обновляет токен и
  тянет `me`; лента показывается из **персиста мгновенно** (cache-first), сеть обновляет в фоне.
  Нет вспышки экрана логина. Нет cookie → страница логина.

## 2. Лента и посты

### 2.1 Просмотр ленты подписок

- **Предусловия:** залогинен.
- **Ожидаемо:** лента постов автора и тех, на кого он подписан, курсорной пагинацией; «Загрузить
  ещё» дописывает страницу плавно (без мигания скелетоном, `startTransition`); дедуп при догрузке.

### 2.2 Создание поста → live-появление у подписчиков

- **Шаги:** автор публикует пост.
- **Ожидаемо:** у автора пост появляется сразу; у **подписчиков с открытой лентой** он
  «прилетает» сверху live (`postAdded` через WS) без перезагрузки, без дублей (дедуп по id).

### 2.3 Поиск по ленте

- **Шаги:** печатает запрос в поле над лентой.
- **Ожидаемо:** ввод плавный (`useDeferredValue`), список фильтруется по загруженным постам с
  лёгким притуханием на время «догона»; несовпавшие карточки **скрываются, не размонтируются**
  (открытая ветка комментариев и черновик переживают фильтр); нет совпадений → «Ничего не найдено».

## 3. Реакции (лайки)

### 3.1 Оптимистичный лайк

- **Шаги:** жмёт ❤️ на посте.
- **Ожидаемо:** сердце заливается и счётчик растёт **мгновенно**, до ответа сервера; тот же пост в
  другом месте (лента ↔ страница поста) показывает то же значение (общая нормализованная запись);
  счётчик не двоится.
- **Негатив:** мутация упала на сервере → Apollo откатывает оптимистичный слой (тост об ошибке —
  отдельная будущая доработка).

### 3.2 Чужая реакция на открытом посте (live)

- **Предусловия:** пользователь открыл страницу поста / развернул ветку.
- **Ожидаемо:** реакция другого пользователя двигает счётчик live (`reactionAdded` по `postId`);
  событие приходит **только на новую реакцию**, смена типа реакции счётчик не двигает; своё эхо
  счётчик не задваивает.

## 4. Комментарии

### 4.1 Оптимистичный комментарий

- **Шаги:** разворачивает ветку (💬), пишет комментарий → «Отправить».
- **Ожидаемо:** «отправляемый» комментарий появляется сразу полупрозрачным (`useOptimistic`),
  затем заменяется реальным; форма сбрасывается сама; счётчик комментариев растёт.
- **Негатив:** запрос ветки не удался → показывается ошибка с «повторить», форма не рисуется над
  пустотой (иначе успешный комментарий молча потерялся бы).

### 4.2 Чужой комментарий на открытом посте (live)

- **Ожидаемо:** комментарий другого пользователя дописывается в ветку live (`commentAdded` по
  `postId`) без дублей; свой комментарий, пришедший и мутацией, и подпиской, не двоится (дедуп по id).

### 4.3 Индикатор «печатает…»

- **Ожидаемо:** пока другой пользователь печатает в этой ветке — «Кто-то печатает…»; сигнал
  эфемерный (мимо кэша), с дебаунсом; если печатающий отвалился, не отправив «стоп» — индикатор
  гаснет по TTL, а не висит вечно.

## 5. Уведомления

### 5.1 Живые уведомления

- **Ожидаемо:** подписка/реакция/комментарий к постам пользователя → колокол 🔔 показывает счётчик
  непрочитанных (считается из кэша), новое уведомление прилетает live (`newNotification`), падает в
  список сверху; рендер полиморфный по типу (`FollowNotification`/`ReactionNotification`/`CommentNotification`).

## 6. Профиль и медиа

### 6.1 Публичный профиль

- **Шаги:** переход по `/u/:username` (клик на автора; при наведении аватар преднагружается).
- **Ожидаемо:** карточка профиля (имя, @username, bio, аватар); заголовок вкладки и `<meta>` =
  имя пользователя (нативные метаданные React 19). Профиль доступен без входа.

### 6.2 Загрузка аватара

- **Предусловия:** пользователь на **своём** профиле.
- **Шаги:** «Сменить аватар» → выбирает изображение.
- **Ожидаемо:** статус «Загрузка…», затем аватар обновляется **во всех местах** (мутация вернула
  `User`, Apollo обновил запись); на бэкенде рядом с файлом фоном появляются `.webp`/`_thumb.webp`.
- **Негатив:** не-картинка или подделка типа (расширение ≠ содержимому) → `400`; файл > 15 МБ →
  быстрый `413` (не зависание); ≥ 10 загрузок/час → `429`. Прошлый аватар удаляется при замене.

## 7. Присутствие (presence)

### 7.1 Онлайн-индикаторы

- **Ожидаемо:** у авторов онлайн — зелёная точка на аватаре; статус живёт на счётчиках соединений
  в Redis (TTL + heartbeat), приходит потоком `presenceChanged`; при смене пользователя набор
  онлайн-id сбрасывается (чужие точки не «протекают» в новую сессию).

## 8. Отдельная страница поста и навигация

### 8.1 Открытие страницы поста

- **Шаги:** «Открыть» на карточке → `/p/:id`.
- **Ожидаемо:** отдельный **ленивый чанк** `PostPage` подгружается (тонкая полоса прогресса на
  время загрузки); тело поста показывается сразу (быстрые поля), комментарии — отдельным запросом
  (замена серверного `@defer`, которого нет на `graphql@16`).

### 8.2 Ленивая загрузка маршрутов

- **Ожидаемо:** профиль/пост/логин/регистрация грузятся отдельными чанками при переходе; стартовый
  бандл — лента + каркас.

### 8.3 Неизвестный URL / устаревший чанк после деплоя

- **Ожидаемо:** неизвестный адрес → «Страница не найдена» **внутри каркаса** (шапка/навигация на
  месте), а не голый экран; если после нового деплоя старая вкладка не может дозагрузить чанк →
  «Приложение обновилось — перезагрузите» с кнопкой перезагрузки.

## 9. Жизненный цикл сессии

### 9.1 Протухший access-токен во время работы

- **Ожидаемо:** при `UNAUTHENTICATED` цепочка Apollo Links тихо обновляет токен (один in-flight
  refresh) и **повторяет** запрос — пользователь ничего не замечает.

### 9.2 Истечение сессии (refresh не удался)

- **Ожидаемо:** сессия чистится **полностью** — токен, in-memory кэш и снапшот в localStorage
  (не только токен!), WS закрывается; редирект на логин. Данные не остаются на устройстве.

### 9.3 Выход и вход другого пользователя на том же браузере

- **Шаги:** пользователь A выходит → пользователь B входит.
- **Ожидаемо:** на logout снапшот кэша A удаляется из localStorage (персист ставится на паузу,
  чтобы отложенная запись не воскресила ключ); вход B стартует с чистого кэша — **B не видит данных A**.

## 10. Офлайн и быстрый старт

### 10.1 Мгновенный старт из персиста

- **Ожидаемо:** при повторном заходе лента/данные рисуются из localStorage до сети (cache-first),
  сеть обновляет в фоне; на login персист включается, на logout — очищается, на повторном login —
  снова включается.
- **Граница:** переполнение квоты localStorage → запись пропускается, последний валидный снапшот
  сохраняется (без перманентного отключения персиста); офлайн-очередь мутаций — вне текущего объёма.

## 11. Устойчивость к ошибкам

### 11.1 Сбой загрузки ленты/поста/профиля

- **Ожидаемо:** suspense-запрос на ошибке бросает → ловит `ErrorBoundary` соответствующего узла
  (лента/пост/профиль изолированы) с кнопкой «Повторить» (для ленты — через `refetch`); падает
  только затронутый блок, не всё приложение.

### 11.2 Централизованный лог ошибок

- **Ожидаемо:** ошибки, пойманные границей (`onCaughtError`) и НЕ пойманные (`onUncaughtError`,
  fatal), проходят через единую точку `report()` (задел под Sentry/аналитику).
