# Changelog

Заметные изменения проекта, новые сверху. Формат свободный: дата — что изменилось и зачем.

## 2026-07-09 — Бэкенд: фон, события и масштабирование

Всё, что не должно блокировать действие пользователя, вынесено в фон.

- **Событийная развязка** (`@nestjs/event-emitter`): сервисы (`PostsService.create`,
  `FollowsService.follow`, `ReactionsService.react`, `CommentsService.create`) больше не пишут
  уведомления и не публикуют в PubSub напрямую — они лишь **эмитят доменные события**
  (`post.created`/`user.followed`/`post.reacted`/`comment.created`, классы-DTO в `src/events/`).
  `emit` (не `emitAsync`) — пользователь не ждёт побочных эффектов; ошибка слушателя изолирована.
- **Очереди BullMQ** (`@nestjs/bullmq`): три очереди — `feed-fanout`, `notifications`,
  `maintenance` — с retry/backoff и `removeOnComplete/Fail` (не копим выполненные в Redis).
  Свои коннекты к Redis из `REDIS_URL` (не передаём наш ioredis-инстанс). jobId без `:`
  (BullMQ запрещает двоеточие) — используем `-`.
- **Fan-out ленты (fan-out on write)** — центральный кейс: `FanoutListener` на `post.created`
  публикует `postAdded` (WS, онлайн-подписчикам) и ставит задачу разноса; `FeedFanoutProcessor`
  разносит id поста по лентам подписчиков (и в ленту самого автора) батчами по 1000;
  `FeedService` хранит ленту в Redis sorted set `feed:<userId>` (score = время поста, обрезка до
  800 свежих, TTL 14 дней). Чтение `feed` теперь O(1) из набора + гидрация постов; **гибрид** —
  при пустом наборе (новый юзер / истёк TTL) бэкофилл из БД с заполнением Redis на будущее.
  Query `feed` переехал из `PostsResolver` в `FeedResolver` (score-курсор вместо id-курсора,
  контракт `PostConnection` для фронта не изменился). «Звёздная» проблема (не разносить посты
  авторов-миллионников, домешивать на чтение) — помечена как расширение.
- **Уведомления через слушатель + очередь доставки:** `NotificationListener` создаёт уведомление,
  публикует real-time событие и ставит доставку в очередь; `NotificationDeliveryProcessor` —
  заглушка SMTP/FCM с **идемпотентностью** (Redis `SET NX delivered:<id> EX 7d`: ретрай не шлёт
  письмо дважды).
- **Планировщик** (`@nestjs/schedule`): `MaintenanceScheduler` по `@Cron` ставит задачи с
  **фиксированным jobId** (`recompute-trending` каждые 30 мин, `daily-digest` в 9:00) — BullMQ
  дедуплицирует по jobId, задача выполняется один раз на любом числе инстансов (jobId =
  распределённый замок). `MaintenanceProcessor` их выполняет.
- **Тренды:** `TrendingService` считает топ-10 хэштегов за 24ч сырым SQL с JOIN (`$queryRaw`,
  параметризован), кэширует JSON в Redis (`trending:hashtags`, TTL 2ч) через уже подключённый
  `REDIS_CLIENT` (без `cache-manager` — меньше зависимостей и версионных нюансов). Query `trending`
  раздаёт из кэша. `DigestService` — каркас рассылки (выборка активных + лог-заглушка).

**По итогам адверсариального ревью (14 находок, 2 HIGH — все исправлены):**

- **Пагинация ленты — тупики и пропуски (HIGH+MEDIUM):** бэкофилл материализовал только `limit`
  постов, а страница 2 (курсор задан) не имела DB-фолбэка → пустая «страница 2», новый подписчик
  застревал на первой; та же причина — тупик за пределами 800 (хвост Redis); score-only курсор без
  id-tiebreak терял посты с одинаковой миллисекундой на границе. Переделал на **гибрид**: первая
  страница — из Redis (быстро), **страницы 2+ и промах — keyset-пагинация по БД** `(createdAt,id)`
  (стабильно, без тупиков и пропусков; курсор несёт полный ключ сортировки).
- **BullMQ-коннекшн терял пароль/db/TLS из `REDIS_URL` (HIGH):** `forRootAsync` брал только
  `{host,port}` → на проде с авторизацией/`rediss://` все фоновые задачи молча не подключались бы.
  Теперь URL разбирается полностью (`username/password/db/tls`).
- **Повторный follow плодил дубли FOLLOW-уведомлений (MEDIUM):** `upsert` был no-op, но событие
  эмитилось всегда. Теперь `create` + `catch(P2002)` — повторный follow не эмитит событие.
- **Unfollow оставлял посты бывшей подписки в ленте (MEDIUM):** материализованный набор не
  чистился. Теперь `unfollow` эмитит `user.unfollowed` → слушатель в FeedModule сбрасывает
  `feed:<userId>` (пересоберётся из БД с актуальным графом).
- **Упавшая cron-задача навсегда блокировала планировщик (MEDIUM):** фиксированный `jobId` +
  удержание проваленной задачи → повторный `add` дедуплицировался в неё. Добавил
  `removeOnFail:true` — каждый cron стартует с чистого `jobId`.
- Отклонено (1): «смена типа реакции не шлёт `reactionAdded`» — не регресс, поведение идентично
  этапу 4 (счётчик-ориентированная подписка публикует только добавление реакции).

Проверено вживую (Docker Postgres+Redis): fan-out материализует ленты подписчика и автора;
бэкофилл + **сквозная пагинация нового подписчика через все страницы** (без дублей/тупика);
реакция/подписка/комментарий рождают уведомления событием; доставка логируется, `delivered:<id>`
ключи выставлены; повторный follow не плодит уведомлений; unfollow чистит ленту; тренды считаются
raw SQL и кэшируются; e2e-подписки 10/10 (через новый путь события→слушатели); typecheck/build зелёные.

## 2026-07-09 — Производительность и архитектура фронтенда

Финальный этап фронта: продакшен-полировка на текущем стеке (без апгрейда graphql до alpha).

- **Code-splitting по маршрутам** (`route.lazy` React Router 7): профиль, страница поста,
  формы логина/регистрации грузятся отдельными чанками — стартовый бандл худеет до ленты +
  каркаса (в сборке видны `ProfilePage-*.js`, `PostPage-*.js`, `LoginForm-*.js`, `RegisterForm-*.js`).
  Главная и `RequireAuth` — в основном бандле (нужны сразу).
- **Страница поста `/p/:id`** (`PostPage`, новый ленивый маршрут; пермалинк «Открыть» в
  `PostCard`): быстрые поля поста через `useSuspenseQuery` показываются мгновенно, ветку
  комментариев отдельным запросом тянет `CommentThread`. Это **осознанная замена серверного
  `@defer`**: у бэкенда `graphql@16`, где директивы `@defer` нет (проверено — `Unknown directive
  "@defer"`), а апгрейд к alpha-графу исключён. Эффект для пользователя тот же — тело поста не
  ждёт комментариев.
- **Персист нормализованного кэша** (`apollo3-cache-persist` → `localStorage`, ключ
  `social:apollo-cache`, потолок 2 МБ, триггер `write` с дебаунсом): восстановление до первого
  рендера (`initCache` в async-bootstrap) — мгновенный старт и офлайн-чтение. На logout —
  `pause()` + `purge()` (иначе отложенный persist от `clearStore` пересоздал бы ключ уже после
  очистки), на login — `resume()`. Совместимость с Apollo Client 4 проверена round-trip'ом
  (в пакете peer заявлен на v3, но работает по утиной типизации `extract`/`restore`).
- **Архитектура ошибок:** `onUncaughtError` (не поймано границей → fatal) и `onCaughtError`
  (перехвачено `ErrorBoundary`) в `createRoot`, единая точка `report()` (задел под Sentry);
  **route-level `errorElement`** (`RouteError`) — падение `route.lazy()` (напр. 404 на чанк после
  деплоя) и неизвестные URL рендерятся внутри каркаса с кнопкой перезагрузки, а не голым дефолтным
  экраном React Router; catch-all `*` → 404; индикатор навигации (`useNavigation`) на время загрузки
  чанка. `errorPolicy` оставлен **дефолтным**: наши suspense-чтения — «всё или ошибка», запрос
  должен бросать, чтобы сработала `ErrorBoundary` с ретраем (глобальный `"all"` сделал бы промис
  resolved с `data:undefined` и убил бы границы — от него отказались после ревью).
- **React Compiler** — проверка через `react-scan` (только dev, динамический импорт до рендера).
- **Federation-развилка** (докой): при уходе бэка в Federation меняется только `VITE_API_URL`
  (gateway), а `VITE_WS_URL` указывает на subgraph с подписками — `split`-линк остаётся.

**По итогам адверсариального ревью (7 находок подтверждено, все исправлены):**

- **Утечка кэша между пользователями (HIGH):** при рантайм-истечении сессии (`onSessionExpired`)
  чистился только токен — снапшот прошлого пользователя оставался в `localStorage` и через
  cache-first протекал следующему на том же браузере. Теперь и logout, и истечение зовут общий
  `clearLocalSession()` (токен + `clearStore` + `purge` + WS), а `login` стартует с чистого кэша.
- **`errorPolicy:"all"` убивал `ErrorBoundary` (3×MEDIUM):** под ним suspense-запрос не бросал,
  а отдавал `data:undefined` → границы мертвы, ошибка молча превращалась в «не найдено»/«недоступна»
  без ретрая. Убрали глобальный `errorPolicy` (вернулись к дефолту) — границы с повтором снова живут.
- **Падение ленивого чанка → голый экран RR (MEDIUM):** добавили `errorElement` + catch-all (см. выше).
- **Двойной счёт своего лайка (MEDIUM):** эхо `reactionAdded` своего действия инкрементило счётчик
  в корневом слое, пока оптимистичный лайк жил в отдельном слое → +2. Для своего эха теперь только
  синхронизируем `myReaction` (идемпотентно), счётчик ведут оптимистичный клик + ответ мутации.
- **maxSize-переполнение навсегда выключало персист (LOW):** библиотечный `maxSize` при превышении
  удалял хороший снапшот и ставил неустранимую паузу. Заменили на `maxSize:false` + свой storage,
  глотающий `QuotaExceededError` (последний снапшот сохраняется, само-восстановление при ужатии кэша).

Проверено вживую через CDP (headless Chrome, прод-сборка): приложение монтируется без ошибок
в консоли, ленивый чанк логина подгружается, лента рендерится, `/p/:id` подгружает чанк
`PostPage`, персист пишется в `localStorage` на login, очищается на logout и восстанавливается
на повторном login; неизвестный URL → «Страница не найдена» внутри каркаса. typecheck/lint/build
зелёные.

## 2026-07-08 — Оптимистичный UX, комментарии и медиа

**Бэкенд (`apps/api`):**

- `Post.comments` — read-path ветки комментариев: field-резолвер + DataLoader
  `commentsByPostId` (один `findMany` на все посты выборки, порядок треда от старых к новым).
- Модуль `media`: мутация `uploadAvatar(file: Upload!)` по спеке GraphQL multipart request
  (`graphql-upload-minimal`); поток на диск через `pipeline` с чисткой недокачанных файлов;
  ключ файла с расширением из mimetype (корректный Content-Type у статики); whitelist
  jpeg/png/webp. Обработка **sharp** в фоне (fire-and-forget): авто-ориентация, ресайз ≤512,
  `.webp` + `_thumb.webp`, EXIF/GPS не копируются — ответ мутации обработку не ждёт.
  С появлением очередей переедет в воркер без смены контракта.
- Раздача загруженного: `/static/` из `storage/` (`useStaticAssets`), helmet CORP ослаблен до
  `cross-origin` (SPA живёт на другом origin), новая настройка `PUBLIC_URL` — база абсолютных
  ссылок на файлы. `graphqlUploadExpress` повешен только на `/graphql` (лимит 15 МБ, 1 файл).
- `reactionAdded` публикуется только при **создании** реакции: смена типа существующей больше
  не даёт события (живые счётчики на клиенте не задваиваются) и не плодит повторных уведомлений.
- Нюанс, зафиксированный в коде: параметр `Upload` типизирован как `FileUpload` (не
  `Promise<FileUpload>`), иначе глобальный `ValidationPipe` (transform) пересоздавал промис
  через `new Promise(undefined)` и ломал загрузку.

**Фронтенд (`apps/web`):**

- Оптимистичный лайк: `LikeButton` на `optimisticResponse` + `cache.modify` (мутации
  `react`/`unreact` возвращают `Boolean` → кэш правится вручную); во фрагмент `PostCard_post`
  добавлен `myReaction`.
- Ветка комментариев: разворачивается по клику на 💬 — ленивый запрос `PostComments`,
  «отправляемый» комментарий через **React 19 `useOptimistic`** (полупрозрачный, откат
  автоматический), отправка через form action (форма сбрасывается сама), сигнал typing с
  дебаунсом переехал сюда из `CommentBox` (удалён).
- Живые события развёрнутого поста: `commentAdded` (чужие комментарии) и `reactionAdded`
  (чужие реакции двигают счётчик через `cache.modify`); общий помощник `appendToCache`
  с дедупом по id — свой комментарий приходит и мутацией, и подпиской.
- Клиентский поиск по ленте: `SearchableFeed` на `useDeferredValue` (ввод отзывчив, список
  «догоняет» с притуханием); в запрос ленты добавлено поле `content`.
- Медиа: терминальный линк `UploadHttpLink` (`apollo-upload-client` v19) вместо `HttpLink` —
  File в переменных уезжает multipart-запросом, заголовок `apollo-require-preflight` для
  CSRF-защиты Apollo Server; в codegen скаляр `Upload → File`; `AvatarUpload` — мутация
  возвращает `User`, Apollo сам обновляет нормализованную запись (аватар меняется везде).
- Страница профиля (`ProfilePage`): `useSuspenseQuery` + нативные метаданные документа React 19
  (`<title>`/`<meta>` из дерева) + `preload` аватара при наведении на автора в ленте;
  `ProfileCard` стал презентационным, показывается `bio`; загрузка аватара — только на своём
  профиле.

**Устойчивость и безопасность (по итогам адверсариального ревью — 24 гипотезы, 19 подтверждено
и исправлено):**

- Загрузка сверх `maxFileSize` больше не подвешивает запрос навсегда: graphql-upload шлёт
  busboy-событие `limit` без ошибки/end у потока — теперь поток рушится явно и клиент
  получает 413 за ~0.1 с; недокачанный файл удаляется.
- Содержимое файла сверяется с заявленным mimetype по **magic bytes** (JPEG/PNG/WebP) до
  записи URL в БД; `uploadAvatar` под rate-limit (10/час); sharp получил `limitInputPixels`
  (25 МП) и таймаут — защита от pixel-бомб; при повторной загрузке старый файл и его
  webp-производные удаляются (и при неудачном апдейте БД новый файл не сиротеет);
  `PUBLIC_URL` обязателен в production (иначе в БД пеклись бы localhost-ссылки).
- WS-подписки: DataLoader'ы контекста живут всю подписку — их кэш теперь сбрасывается
  перед каждым событием (`freshLoadersPerEvent`), иначе поздние события отдавали бы
  устаревших авторов и счётчики. Реакция создаётся через `create` → на `P2002` `update`:
  уникальный индекс сам решает гонку, событие/уведомление не задваиваются. Порядок
  комментариев стабилен при равных `createdAt` (тай-брейкер по id).
- Фронт: ветка комментариев на `cache-and-network` (повторное открытие дотягивает
  пропущенное), при упавшем запросе — ошибка с ретраем вместо формы над пустотой
  (успешный `addComment` без поля `comments` в кэше молча терялся: `cache.modify` не
  создаёт полей); поиск по ленте скрывает карточки, а не размонтирует (развёрнутые ветки
  и черновики переживают фильтр); свои реакции из другой вкладки синхронизируются
  (не отбрасываются как эхо); редьюсер `useOptimistic` чист (id генерируется в submit);
  «призрак» своего комментария прячется, когда WS-копия обгоняет ответ мутации;
  индикатор «печатает…» с TTL (обрыв у печатающего не вешает его навсегда) и очисткой
  таймеров/сигнала при размонтировании; `isTyping:true` шлётся раз на серию набора,
  а не на каждую клавишу.

Проверено вживую: multipart-загрузка end-to-end (webp-производные появляются, статика отдаёт
корректный Content-Type и CORP), oversize → мгновенный 413, подделка сигнатуры → 400,
замена аватара подчищает старые файлы, троттлер отдаёт 429, `reactionAdded` не срабатывает
на смену типа (WS-клиент), `Post.comments` с DataLoader, e2e-сьюты бэка 10/10,
typecheck/lint/build обоих приложений.
